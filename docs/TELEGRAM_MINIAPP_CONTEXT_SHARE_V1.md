# GBot Telegram Mini App Context & Share Card Spec V1.5

本规范定义了 GBot 在 Telegram Host 容器下的上下文解析原则与战报分享机制的安全规范。

## 1. Telegram Mini App Context 运用原则
- **安全隔离**: Agent 不暴露或在网络上传输 Telegram `initData` 的敏感原始哈希，防止中间人伪造请求。
- **UI 显示限制**: Telegram 用户 ID 只能以缩写形式（如 `123...789`）在非关键调试面板中预览，绝不在任何公开界面泄露。
- **宿主自适应**: 前端基于 `window.Telegram.WebApp` 的可用性自动调节展示界面：
  - 在 Telegram 容器内运行：显示真实连接参数、平台信息（Android/iOS/Desktop/MacOS）及用户昵称。
  - 在一般浏览器运行：优雅回退并提示“等待授权连接”。

## 2. 战报分享卡模型 (Work Report Share Card Model)
- 战报分享由用户点击“复制战报摘要”或未来的 Telegram Native Share 接口调起。
- **信息规范**:
  * 必须指明 Agent 的具体名称与成长等级。
  * 必须客观展现任务消耗的 AI Credits。
  * 必须增加核心合规免责声称：“任务结果取决于任务方验收与结算，不提供任何收益或产出承诺”。
  * 绝对禁止虚报假大空的预期获利或对结果作确定性承诺。

## 3. 数据边界与权限规范
- **禁止私聊监控**: Agent 没有且绝不申请私聊对话记录的后台监听权限。
- **禁止全群消息读取**: Agent 在群聊公会环境只能识别以 `@GBot` 明确唤起的命令或者由用户/群管理员主动提交的线索，绝对不具有隐式读取全部聊天流的历史记录。
- **禁止自动私信与刷屏**: 严禁利用 Agent 进行批量的陌生人私发广告或在群组内高频发送无关内容，坚守 Telegram 防垃圾信息红线。
- **禁止无授权网络抓取**: 不利用第三方登录授权进行全平台的垃圾信息分析与未经允许的 AI 模型训练数据收集。

## 4. 未来后端集成需求 (Future V2 Scope)
- 用户授权数据与已绑定群聊的白名单需加密存储于后端数据库。
- 只有被 Policy Guard 审查并符合主人每日额度的 `OnchainTransactionIntent` 才可以向链上执行。
- 引入群组声望和每周排行榜的数据缓存一致性审计。
