# GBot Bubble Agent 商业资产与技能系统设计规范 (V2)

本规范在 V1 资产模型基础上，正式引入了“**普通泥泡泡 + 五款特别版泥泡泡**”的商业分类模型，并整合了“**天生内置技能 + 4个可装备槽**”的技能闭环系统与“**技能盲盒**”配置机制，规避金融合规红线，赋予气泡召唤兽可持续发展的应用内商业与生态价值。

---

## 1. 资产双层与分类模型

### 1.1 双层钱包模型
- **应用内 Agent 资产（默认）**：存于中心化数据库，官方发售/发放的均为此类，支持升级、装配技能、配饰及展示编号。不可直接链上交易。
- **链上 Agent Passport（用户自愿铸造）**：用户支付 Gas 费将应用内资产铸造为 NFT，铸造后生成具有唯一 `token_id` 的链上 Passport，允许在外部二手市场展示与转赠。

### 1.2 泡泡商业分类定位
系统通过“主线普通气泡”与“特别版气泡”进行商业差异化，所有泡泡**均不挂钩任何代币收益回报率**：

| 召唤兽系列 | 颜色谱系 | 稀有度 (Rarity) | 槽位与技能规则 | 商业定位与生态作用 |
| :--- | :--- | :--- | :--- | :--- |
| **普通泥泡泡** | 烟灰/石墨灰 | Common (默认) | 0 个天生技能 + 4 个可装配槽 | **大众基础款**。所有用户注册默认获得，具备基础养成与技能升级路径。 |
| **黑金泥泡泡** | 主线黑金渐变 | Rare / Epic | 1 个天生高级策略标签 + 4 个可装配槽 | **经典特别版**。经典高雅形象，初始携带“冷静模块”高级标签。 |
| **冰蓝泥泡泡** | 水流浅蓝透明 | Rare / Epic | 1 个天生高级策略标签 + 4 个可装配槽 | **分析偏向款**。初始携带“数据镜片”高级标签，偏向数据化展示。 |
| **暗紫泥泡泡** | 幽邃暗紫核心 | Rare / Epic | 1 个天生高级策略标签 + 4 个可装配槽 | **策略偏向款**。初始携带“暗线观察”高级标签，偏向链上信号叙事。 |
| **赤金泥泡泡** | 熔岩红色渐变 | Rare / Epic | 1 个天生高级策略标签 + 4 个可装配槽 | **执行偏向款**。初始携带“快速回声”高级标签，偏向节奏感展示。 |
| **白银泥泡泡** | 液态水银金属 | Genesis (创世纪念) | 1-3 个天生纪念标签 + 4 个可装配槽 | **限量创世纪念系列**。天生携带“Genesis 印记”专家级标签，用于身份专属展示。 |

> [!NOTE]
> **默认发放与派生规则**：所有新注册用户默认发放 Common 烟灰/石墨灰普通泥泡泡，且初始无天生技能。特别版泥泡泡（如黑金、冰蓝、暗紫、赤金和白银）只能来自于后续的盲盒开出、特定运营活动或官方限量分发。**绝对禁止前端使用哈希函数随机将普通用户默认泡泡派生为特别版形象**。

### 1.3 前端动画资源约定
- 当前 miniapp 使用 Canvas 2D 序列帧播放泥泡泡形象，资源目录固定为 `apps/miniapp/public/agent-bubble-dark/v3/<colorGene>/`。
- 已支持的 `colorGene` 目录包括：`gray`、`black-gold`、`blue`、`purple`、`red`、`silver`。
- 每个颜色目录必须包含同名动作表：`idle-slime-sheet.png`、`tap-round-sheet.png`、`tap-square-sheet.png`、`tap-triangle-sheet.png`、`dispatch-sheet.png`、`busy-sheet.png`、`reward-sheet.png`、`waiting-sheet.png`、`tired-sheet.png`、`failed-sheet.png`。
- 点击交互采用单次点击单次变形循环：第一次圆形、第二次方形、第三次三角形，然后循环。禁止恢复一次点击连续播放多形态。
- 主按钮“派 Agent 赚钱”采用独立 `dispatch` 出勤动作：泥泡泡先压低并拉成长条胶囊态，再衔接 `busy` 执行态。`dispatch` 只表示任务状态反馈，禁止被包装为收益承诺或成功率暗示。
- 视觉总览图输出为 `apps/miniapp/public/agent-bubble-dark/v3/bubble-variants-overview.png`，用于设计与运营验收，不作为前端运行时依赖。

## 2. 技能系统与槽位规则 (Skills System)

为了最大化召唤兽的养成趣味与可玩性，建立如下技能规则：

### 2.1 技能分类
- **普通技能 (Common)**：执行基础层任务扫描。
- **高级技能 (Advanced)**：天生携带或从高级盲盒获取，策略广度更大。
- **专家技能 (Expert)**：极低概率开出，具备专家级任务辅助动作。

### 2.2 技能来源
1. **用户购买技能卡**：在游戏内商城以 G 币或 TON 币购买特定普通/高级技能卡。
2. **技能盲盒开出**：通过盲盒系统随机抽取技能道具。
3. **特别版泡泡天生携带**：特别版和创世纪念版天生固化携带特定天生技能。

### 2.3 槽位装配规则
- **可装配槽**：所有气泡（包括普通和特别版）一律固定为 **4 个可装配槽**。
- **天生技能（不占槽）**：
  - 特别版泡泡额外具有 **1-3 个天生技能**。
  - 天生技能**不占用** 4 个自由装配槽位。
  - 天生技能**永久固化不可拆卸**，不随资产转移而消失，直接显示于身份铭牌或档案卡。
- **合规边界**：天生技能仅扩展了气泡在游戏内策略的选择广度与动作/战报展示风格，**绝对禁止出现“套利”、“高频”、“提升收益”、“加持”、“捕获瞬时机会”或“执行深度”等收益性与对抗性高风险承诺词，不承诺任何代币收益增幅或成功率，强调展示差异与战报趣味性**。

## 3. 技能盲盒配置机制 (Blind Box)

盲盒概率与内容由后台服务端进行配置与权重管理，以确保概率的随时调控与合规。

### 3.1 盲盒池数据模型 (TS/Go 结构)
```typescript
interface BlindBoxPoolItem {
  itemId: string;
  itemType: "skill" | "bubble_agent" | "cosmetic" | "motion_pack" | "frame";
  rarity: "Common" | "Rare" | "Epic" | "Legendary" | "Genesis";
  weight: number;      // 相对权重，计算概率公式：weight / sum(weights)
  enabled: boolean;     // 是否启用
}
```

### 3.2 概率分配与审计机制
- **前端概率降级**：盲盒概率和池子构成不得在前端硬编码或作为正式的判定源。前端代码中仅声明示例项配置（如 `MOCK_BLIND_BOX_POOL_EXAMPLE`）。
- **后端统一控制**：实际抽卡的结算与权重配置必须完全由后端管理系统或智能合约逻辑控制。客户端展示的概率数据必须从后端接口读取并比对一致，任何特别版概率的调整必须在后台数据库或日志中留下详细的操作记录以便合规性审查。
- 盲盒支持物品类型涵盖：普通/高级/专家技能卡、特别版气泡、配饰、特有动作包、限定舞台背景框。

### 3.3 Mini App 前端闭环页面规则

当前 Mini App 的运营闭环以四个游戏化页面承接：

1. **Agent 主屏**：只展示 Agent 形象、编号铭牌、资源状态条、已装配技能、派遣主按钮、今日简报和最近战报。禁止重新堆叠成长仪表盘。
2. **去赚钱 / 任务地图**：用户点击“派 Agent 赚钱”后进入任务地图。页面必须优先展示路线节点、Token 能量消耗、候选 G、风险确认状态和战报入口。候选奖励必须描述为“待验收 / 候选 / 待结算”，不得描述为已保证到账。
3. **技能商店 / 技能盲盒**：统一承接技能卡购买、盲盒说明和特别版泥泡泡图鉴。特别版泥泡泡可作为盲盒或活动结果展示，但正式概率、库存和启停必须由后台控制。
4. **小金库 / 公会**：小金库展示 Agent 自己的钱包、预算、Token 能量、背包、Passport 自愿铸造状态；公会展示泡泡编号、战报和邀请传播，不承诺任何资产升值或固定式回款。

上述页面均必须保持移动端优先、傻瓜式操作，不得把首页或任务页恢复为后台数据面板。

### 3.4 特别版泡泡商业闭环

- **默认发放**：所有新用户获得 Common 烟灰泥泡泡，初始无天生技能，具备 4 个可装配槽。
- **特别版来源**：特别版泡泡只来自技能盲盒、活动、运营配置或纪念发放，不允许前端用随机哈希为普通用户派生特别版。
- **特别版价值点**：特别版泡泡的差异来自外观、底框、动作包、编号展示和 1-3 个天生技能标签。天生标签不占 4 个可装配槽。
- **技能复投**：用户通过任务获得候选 G，经验证和结算后可继续购买技能卡或参与技能包，形成“派遣 -> 战报 -> G -> 购买技能/特别款 -> 更丰富任务展示”的循环。
- **上链原则**：官方发放/售卖的泡泡默认是应用内资产。只有用户主动发起 Passport 铸造并完成链上确认后，才会成为可在外部展示或转赠的链上资产。

### 3.5 运营配置 V1 与后台预览页

当前 V1 运营配置采用“服务端只读 API 优先，shared 静态配置兜底”的模式。配置 API 只负责把运营口径统一分发给 Mini App 与 Admin，不负责真实抽取判定：

- `packages/shared/src/index.ts` 中的 `GBOT_BUBBLE_EDITIONS` 定义普通款与五款特别版泡泡的发行属性、来源、底框和天生标签。
- `GBOT_BUBBLE_BLIND_BOX_POOL` 定义技能盲盒奖池条目、类型、权重、启停状态与说明。Mini App 可以展示权重预览，但正式抽取、库存和启停必须由后端控制。
- `GBOT_BUBBLE_PASSPORT_SYNC_PREVIEW` 定义 Passport 同步状态示例，用于说明应用内资产、自愿铸造、链上确认、索引同步和待处理状态。
- `GBOT_BUBBLE_SHARE_SURFACES` 定义首页编号泡泡、任务战报、泡泡图鉴、公会展示墙、Passport 铭牌等分享位。
- `GBOT_BUBBLE_AGENT_OPS_CONFIG` 聚合上述配置，作为 API fallback 与本地预览的唯一静态源。
- API Worker 暴露 `GET /bubble-agent/config` 给 Mini App 读取，并暴露 `GET /admin/bubble-agent/config` 给 Admin 读取。

Admin 后台新增 `泡泡运营 V1` 只读配置页，承接运营侧预览：

1. **摘要卡**：展示泡泡版本数、启用盲盒条目、总权重、天生标签数量、Passport 状态样例。
2. **运营边界**：明确所有用户默认获得 Common 烟灰泥泡泡；特别版来自技能盲盒或活动；官方泡泡默认不是链上资产。
3. **技能盲盒概率池**：展示 API/shared 权重和预览概率。此页面不负责抽取判定，不提供伪保存。
4. **泡泡形象发行表**：展示颜色谱系、稀有度、来源、底框与天生标签。
5. **Passport 同步预览**：展示 `unminted`、`minting`、`minted`、`failed` 等状态说明。
6. **分享入口配置**：展示可传播的位置和默认文案，后续可接后台 A/B 文案配置。

Mini App 对应承接点：

- Agent 主屏提供“分享泡泡”次级入口，并记录分享事件。
- 技能商店展示盲盒启用条目、总权重、特别泡泡数量与奖池预览。
- 技能商店提供“开一个技能盲盒”入口。真实模式复用后端 `POST /store/boxes/:boxId/orders` 购买与 `POST /boxes/:inventoryItemId/open` 开盒流程；前端只展示返回结果，不做概率判定。
- 小金库展示 Passport 同步状态说明，避免用户误解官方泡泡默认进入链上流程。
- 公会页提供展示墙分享入口，形成“编号泡泡 -> 战报 -> 公会展示 -> 邀请回流”的传播闭环。

### 3.6 技能盲盒开出结果契约

`BubbleBoxOpenResult` 是 Mini App 展示层与后端开盒流程之间的统一展示契约：

- `type=skill`：技能卡结果，进入背包后可被学习或装配。
- `type=bubble_agent`：特别版泡泡结果，带 `bubbleEditionKey`、`displayNo` 和 `naturalSkillCodes`，展示为应用内泡泡资产。
- `type=motion_pack/frame/cosmetic`：外观动作或底框消耗品。
- `type=pending_points/energy/ability`：兼容现有旧开盒流程的奖励类型。

正式开盒必须由后端事务完成：扣减预算、生成订单、写入背包、烧毁盲盒、写入审计事件和返回结果。前端不得自行随机生成正式结果；Mock 模式只用于本地演示。

### 3.7 泡泡背包与当前展示规则

特别版泡泡进入背包后，不应和普通消耗品混在同一层级展示。Mini App 小金库需要提供独立的“我的泡泡资产”区域：

- 泡泡资产来自 `InventoryItem`，通过 `bubbleEditionKey`、`displayNo`、`series`、`naturalSkillCodes` 等字段识别。
- 用户可以选择一只特别版泡泡作为当前展示泡泡；该选择只影响 App 内首页形象、铭牌、系列和天生标签展示。
- 如果用户没有特别版泡泡，首页默认展示 Common 烟灰泥泡泡。
- 选择当前展示泡泡不代表 Passport 铸造，不改变链上归属，不产生外部交易能力。
- 泡泡资产保留 4 个可装配技能槽；天生标签只作为发行属性和展示属性，不占用装配槽。

### 3.8 Passport 自愿铸造交互流程

Passport 是用户主动发起的链上身份流程，不是官方发放泡泡时默认附带的状态。当前 V1 完成前端交互契约、后端状态持久化和 Admin 只读监控，不接真实钱包签名、不生成真实链上资产。

Mini App 小金库中的 Passport 卡片必须遵循以下状态流：

1. `unminted`：展示“游戏内资产”，按钮为“自愿铸造”。用户点击前不进入链上流程。
2. `minting`：展示“铸造同步中”，按钮禁用为“同步中”。文案说明等待钱包确认、链上确认与索引同步。
3. `minted`：展示“已铸造 Passport”，显示 `token_id` 或待同步编号。按钮禁用为“已铸造”。
4. `failed`：展示“铸造未完成”，按钮为“重新发起”。不得把失败表述为资产损失或收益损失。

实现边界：

- `POST /bubble-agent/passport/mint` 只表示后端已记录用户的自愿铸造请求，返回 `minting` 中间态。
- `GET /bubble-agent/passport/:displayNo` 用于 Mini App 查询当前 Passport 状态。正式态优先读取该接口；Mock 态保留本地调试能力。
- `GET /admin/bubble-agent/passports` 用于 Admin 查看 Passport 当前状态、状态计数和最近事件，只读展示，不允许管理员代替用户发起铸造。
- D1 表 `bubble_agent_passports` 保存每只编号泡泡当前 Passport 状态；`bubble_agent_passport_events` 保存请求与同步事件。
- Mock 模式可用本地状态模拟 `minting -> minted`，用于产品验收；正式态必须以服务端、钱包和链上索引结果为准。
- 首页编号铭牌和小金库 Passport 卡片必须读取同一 `mintStatus` 展示源，避免一个页面显示已铸造、另一个页面仍显示游戏内资产。
- 选择“当前展示泡泡”与 Passport 铸造完全解耦。用户可以展示特别版泡泡，但这不改变链上归属状态。
- 同步时间不得写成固定秒数；只能描述为取决于钱包确认、目标链确认速度与索引服务状态。

## 4. 资产身份稳定性红线 (Asset Identity Stability Red Lines)

为了保证游戏公平性、保护玩家收藏价值及应对 Web3 金融合规要求，确立以下身份红线规则：

> [!IMPORTANT]
> **红线 1：编号永久固定 (Stable display_no)**
> 资产展示编号 `display_no` 必须在生成时由 UUID 稳定哈希决定。**绝对禁止**随召唤兽的等级（Level）提升、属性洗练或装备变化而发生编号变更。
> 
> **红线 2：稀有度与等级解耦 (Rarity Decoupling)**
> 资产稀有度 `rarity` 由角色生成时的天赋 Hash 决定。**绝对禁止**与角色等级挂钩，或随着玩家不断升级、喂养而自动“进化”稀有度等级。
> 
> **红线 3：Genesis 身份恒定 (Genesis Invariance)**
> 创世纪念系列等特定系列属性，必须在生成时即硬编码绑定，**绝对禁止**将后期升级或成就行为作为获得创世身份的手段，以防违背资产稀有度公平性与代币化合规要求。
> 
> **红线 4：发行属性的不可变性与自主铸造原则 (Immutability & Voluntary Mint)**
> 气泡的 `display_no` (展示编号)、`rarity` (稀有度)、`series/colorGene` (系列与色彩谱系) 以及 `Genesis` (创世纪念身份) 均属于原生**发行属性**。发行属性在生成时硬编码固化，**绝对禁止**随着气泡等级升级、技能槽装配、任务战报结果或任何喂养行为而发生变化。官方默认发放的是应用内的 Common 烟灰泥泡泡（默认不进入链上流程，不带金融属性），链上 Passport 必须由用户主动、自愿发起铸造。

## 5. 合规文案红线 (Compliance Copywriting Guidelines)

### 🚫 禁用词与表达 (Prohibited)
*   **收益率承诺**：不得使用高年化确定性回款、每天确定到账、升级越快收益越高等表达。
*   **直接理财属性**：“购买低编号能实现 100% 溢价变现”、“泡泡属于高流动性金融投资品”。
*   **强制描述上链**：“所有泡泡买入即自动成为链上 NFT”。

###  推荐词与表达 (Recommended)
*   **游戏性描述**：“特别版天生携带高级技能，初始策略空间更大、可玩性更强”。
*   **收藏表达**：“天生外观差异更明显，适合进行个性化舞台展示与截图分享”。
*   **铸造描述**：“可由用户在游戏内自愿发起链上铸造 Passport”。
